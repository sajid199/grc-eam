01

Firefighter ID not visible in GRC

The most common “gotcha” question — it's almost always a synchronization gap, not a config error.

Q1I created a Firefighter ID in the connector system, but I can't find it when I search in GRC. What's the likely cause?

Firefighter IDs are created as service IDs directly in the connector (plug-in) system, so GRC won't know about a new one until a sync brings it across. Run a Repository Object Sync (incremental first, then a full sync is recommended) so the connector's user master data lands in GRC.

Sync alone isn't enough, though — the service ID also has to carry the role named in parameter 4010 (standard: GRAC_SPM_FFID). Without that role assigned in the connector system, GRC has no way to recognize the ID as a firefighter at all, even after a successful sync.

Two checks, in order1) Has the repository sync run since the ID was created? 2) Does the ID actually hold the 4010 identifier role in that connector?
02

Architecture — Centralized vs. Decentralized

The opening question in almost every EAM round — get the mental model right before anything else.

Q2What's the difference between centralized and decentralized firefighter?

It's about where the firefighter user logs in first. In a centralized setup, the firefighter user logs into the GRC system itself and launches the Superuser Privilege Management transaction there; GRC then shows the firefighter IDs available to that user across the connected backend systems, and hands off into whichever connector is needed.

In a decentralized setup, the firefighter user logs directly into the connector (backend) system — not GRC — and launches the local firefighter transaction there, which shows only the IDs available in that specific system.

One-line answerCentralized: log into GRC, then reach the connector via the FFID. Decentralized: log straight into the connector and use the FFID locally.
Q3What's the difference between ID-based and role-based firefighter?

In the ID-based model, a dedicated firefighter service ID is created and temporarily assigned to whichever firefighter user needs elevated access — this is the far more common pattern.

In the role-based model, there's no separate service ID; instead a firefighter role carrying the elevated authorizations is assigned directly to the user's own ID when the need arises.

03

Parameters & Configuration

The 4000-series parameters and the authorization footprint interviewers probe for depth of hands-on config work.

Q4Which parameter switches the system between centralized and decentralized firefighter?

Parameter 4015, maintained in SPRO, controls this. It's a single toggle that decides whether the landscape behaves as centralized or decentralized firefighter.

Q5Which parameter tells GRC how to recognize a Firefighter ID, and what goes into it?

Parameter 4010 holds the name of a role — the standard delivered role is GRAC_SPM_FFID. Whatever role is entered here must be assigned, in every connector system, to each service ID that should function as a firefighter.

Practically: if GRC is linked to three or four backend systems, that same identifier role has to exist in each of them and be assigned to the relevant service IDs — GRC only recognizes an ID as a firefighter because it's carrying this role, nothing else marks it as one.

Q6Across multiple connectors, does the 4010 role name have to be identical, or can it vary?

It doesn't have to be identical — SPRO settings allow different identifier role names per connector — but keeping one common role name across all centralized connectors is the accepted good practice, purely for ease of identification.

Decentralized landscapes tend to break from that pattern anyway, since each connector's firefighter concept is managed more independently and often uses its own role name.

Q7What authorization object sits inside the Firefighter ID identifier role?

Just one: S_RFC. The role's job is purely to mark the ID as a firefighter for GRC's recognition — it isn't where the elevated business authorizations live.

04

Owner & Controller Governance

Segregation-of-duties logic behind who can approve, who can review, and where the guardrails sit.

Q8What's the difference between a firefighter owner and a firefighter controller?

The owner approves the assignment of a Firefighter ID to a firefighter user. The controller reviews the usage logs generated once that ID has been used, and signs off on them.

Q9Is a controller mandatory for every Firefighter ID?

No — a Firefighter ID can technically exist without a controller assigned. Whether that's allowed in practice is governed by a related SPRO parameter setting, so it's a policy choice as much as a technical one.

Q10Can the same person be both the owner and the controller for an ID?

Technically yes, but it isn't recommended — it undermines the segregation between approving usage and independently reviewing it, which is the entire point of splitting the two roles.

Q11Can an owner or controller raise the access request for the very Firefighter ID they own or control?

Strictly, no — an owner shouldn't be the one requesting the ID they themselves own, and this can be actively blocked through SPRO parameter settings rather than left to policy alone.

Q12How does a controller actually review the firefighter logs?

The controller receives the log request either as a notification or through an MSMP workflow item sitting in their inbox. Opening it shows every transaction the firefighter user executed under that ID, which the controller then reviews and signs off on.

Q13Can a controller reject a firefighter log?

No — logs can't be rejected outright. If something needs clarification, the controller can only send it back to the firefighter user to explain the usage, not reject it as a formal action.

05

Assignment & Lifecycle

Day-to-day operational questions — how IDs actually get into a user's hands, and for how long.

Q14A firefighter user ran transactions, but the controller never got a notification or a review workflow item. What could be wrong?

Work through it as a short checklist rather than one guess:

1. Log sync job — the firefighter log sync job may not have run; regenerate/run it so the logs actually get pulled into GRC.
2. Workflow trigger — confirm the review workflow actually fired and routed to the controller.
3. Controller assignment — check the controller is genuinely assigned to that specific Firefighter ID.
4. Access & notification config — verify the controller has access to view/approve the request, and that they're correctly maintained for notifications in the MSMP workflow for the log review process.

Interviewer signalNaming "log sync job" alone is a shallow answer — walking through all four checks shows you've actually supported this in production.
Q15In how many ways can a Firefighter ID be assigned to a firefighter user?

Two ways: (1) via a GRC access request, raised through the standard access request workflow set up in MSMP, or (2) via direct admin assignment, done through the GRC administration screens in NWBC without going through a request.

Q16For how many days can a Firefighter ID be assigned to a user?

There's no fixed number — it's project-specific, commonly anywhere from a single day up to a month depending on organizational policy. The default validity period itself is also configurable through parameters.

Q17What kinds of activity do firefighter logs actually capture?

This is configurable via parameters — logs can be scoped to just transactional activity, or extended to also pull in OS-level logs, database-level logs, and similar system logs depending on what the parameter settings enable.

The End-to-End Process

The question that decides the interview — walking through the full firefighter build from a blank system.

2

3

4
Create FFID, FF user, owner, controller, reason codes
5
Sync FFIDs from connector to GRC
6

7

Q18Walk me through the complete Firefighter / EAM setup process, end to end.

Prerequisites first. Configure the 4000-series parameters — decide role-based vs. ID-based, set the 4010 identifier role, set 4015 for centralized/decentralized. Activate the required background jobs: repository sync, firefighter workflow jobs, and firefighter log sync jobs. Build the MSMP workflows for the FFID access request and for the firefighter log review process.

Five key elements to create. Firefighter ID — a service ID created in the connector system with elevated access, scoped per module (finance, procurement, sales & distribution each get their own FFID, not one blanket ID with full access). Firefighter user — the dialog user who actually logs in and uses the FFID, sitting in GRC for centralized or in the connector for decentralized, and holding the standard SPM user role. Firefighter owner and Firefighter controller — created in GRC and assigned the standard SPM owner and SPM controller roles respectively. Reason codes — created in GRC and scoped per connector, not shared globally across all connectors.

Then the operational sequence. Run the repository sync so FFIDs (and, for decentralized, FF users) land in GRC. Once visible, go to Access Control Owners in NWBC and mark the individuals as firefighter owner/controller. Assign that owner and controller to the specific FFID — this has to happen before the ID can be handed to a user. Assign the FFID to the firefighter user, either via a GRC request or admin assignment. The user logs in and executes transactions. Run the FF log sync job so the logs are pulled through, and only then does the controller receive them for review.

Structure to lead withPrerequisites → five key elements → sync → owner/controller marking & assignment → FFID-to-user assignment → usage → log sync → controller review. Naming this sequence, in order, is usually what separates a strong answer from a vague one.
07

User Administration & Housekeeping

General SAP Security fundamentals — user types, locks, mass maintenance, and the tables interviewers expect you to know cold.

Q19What user types exist in SAP, and which type is a Firefighter ID?

SAP has five user types: Dialog, Service, System, Communication, and Reference. Dialog and Service allow interactive GUI logon; System, Communication, and Reference do not.

A Firefighter ID is created as a Service user — that's the standard answer interviewers are listening for.

Q20A Firefighter ID is a service user, which normally allows GUI logon — how would you restrict GUI logon on it specifically?

This is a genuinely tricky follow-up that catches people off guard in the interview. Service users are GUI-logon-capable by default, but a service ID can still be hardened against interactive logon through additional restriction steps at the user/logon-security level, on top of the usual service-user setup.

Honest answer beats a guessIf you haven't implemented this yourself, it's fine to say so and describe the direction (restricting interactive logon capability on an otherwise GUI-enabled service user) rather than inventing exact steps.
Q21Why not make the Firefighter ID a System user instead of a Service user?

In practice, Firefighter IDs are consistently built as Service users, not System users — System users don't support the interactive logon a firefighter user needs when they pick up the ID and work in the system. It's a reasonable question to flag as uncommon/unused in real projects if you haven't seen a System-user implementation of it.

Q22What are the different user lock status values in SAP?

These show up as numeric codes on the user master (in table USR02):

0 Unlocked  ·  32 Locked by CUA  ·  64 Locked by the system administrator  ·  128 Locked due to repeated incorrect logon attempts  ·  192 A combination — locked due to incorrect logons and by the administrator/system.

Q23How do you find which users are currently active in the system?

Query table USR02 — it holds lock status, validity dates, and last logon information per user. To get "active as of today," filter for lock status 0 (unlocked) and a validity end date greater than or equal to today's date.

Q24You have a list of user IDs — how do you retrieve their email addresses?

The quick operational route is SU10 → Address Data tab → feed in the user IDs and execute; it returns the email addresses directly. That's the practical, day-to-day answer.

But interviewers are usually fishing for the table-level route: email addresses live in ADR6, which is keyed by personnel/address number rather than user ID. So the lookup is two-step — first pull each user's personnel number from USR21 using the user ID, then feed that personnel number into ADR6 to get the email address.

Q25How can you lock all users in the system at once?

Two options: mass-maintain and lock everyone through SU10, or use transaction EWZ5, which is purpose-built for mass locking/unlocking of all users in one go.

Q26Where would you check for illegal or rejected passwords?

Table USR40 holds the list of illegal/prohibited password patterns configured in the system. It's a less commonly asked table, but worth knowing by name.

Q27What's the difference between SAP_ALL and SAP_NEW?

SAP_ALL grants authorization to every authorization object in the system. SAP_NEW is narrower and specifically covers new authorization objects introduced during a system upgrade — it's generated so that existing roles don't unexpectedly fail checks against objects that didn't exist before the upgrade.

Q28How do you generally handle user termination and onboarding?

Fair to open with: this is always client- and project-specific, so describe the common pattern rather than claiming one universal process. For termination: lock the user and end their validity date as soon as the exit is confirmed, then a scheduled background job typically removes the assigned roles automatically after a defined grace period — often 30 or 60 days. Onboarding mirrors this in reverse, driven by whatever access-request workflow the client has standardized on.

Q29You need to assign 1,000 roles across 500 users — how do you approach it, especially if the roles aren't the same for every user?

If the same set of roles applies to all the users, SU10 handles it in one mass-maintenance run. The moment the role assignments differ per user — which is the realistic scenario at this scale — SU10 alone won't cut it, since it applies one common role list per run.

The fallback there is a scripted approach: build an eCATT (or similar) script that takes each user paired with their specific role list as input and executes the assignments in bulk, rather than doing 500 individual manual assignments.
